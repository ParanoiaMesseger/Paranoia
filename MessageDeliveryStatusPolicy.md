# Политика статусов доставки сообщений

## 1. Обзор

Политика описывает четыре состояния жизненного цикла исходящего сообщения и механизм хранения `last_seq` на сервере для реализации уведомлений о прочтении.

**Принципы:**

- Сервер хранит только `last_seq` — максимальный `seq`, запрошенный участником диалога при последнем pull. Никакой явной связи `message ↔ delivered/read` на сервере нет.
- Вывод о статусе сообщения делает клиент-отправитель локально: `msg.seq <= partner_last_seq` → прочитано.
- `/arrived` работает **только для активного (открытого) диалога** — не в фоне.
- Opt-out управляется per-dialogue и означает лишь то, что сервер прекращает обновлять `last_seq` для данного диалога. Уже накопленное значение остаётся.

***

## 2. Состояния сообщения

### 2.1. Модель переходов

```
[PENDING] ──pull-ack──► [DELIVERED] ──/arrived──► [READ]
    │
    └── ошибка / таймаут ──► остаётся PENDING (retry backoff)
```

### 2.2. Таблица состояний

| Состояние | Иконка | Условие перехода | Источник данных |
|-----------|--------|-----------------|-----------------|
| `PENDING` | ⏱ | Сообщение добавлено в очередь отправки | Локально |
| `DELIVERED` | ✓ | Сервер вернул `seq` в pull-ответе | pull-ответ сервера |
| `READ` | ✓✓ | `partner_last_seq >= msg.seq` из `/arrived` | GET /arrived |

> Статус вычисляется клиентом: для каждого сообщения диалога проверяется `msg.seq <= partner_last_seq`. Сервер не знает, какое именно сообщение считается прочитанным.

***

## 3. Подробное описание состояний

### 3.1. PENDING — сообщение в очереди

**Триггер:** пользователь нажал «Отправить».

**Действия клиента:**
- Сообщение шифруется и подписывается (`Ed25519`).
- Запись добавляется в локальную SQLite-очередь со статусом `pending`.
- В UI показывается иконка ⏱.
- Клиент выполняет `push` на сервер.

**Ошибка отправки:** статус остаётся `pending`; применяется `exponential backoff + jitter` (см. NotificationsPolicy § 3.4).

***

### 3.2. DELIVERED — доставлено на сервер

**Механизм (pull-based acknowledgment):**

1. Отправитель выполняет `push`.
2. Клиент отправителя выполняет собственный `pull` с `next_send_seq` / `last_pulled_seq + 1` (MultiDevicePolicy).
3. Сервер возвращает в ответе `seq` принятого сообщения.
4. Клиент сопоставляет возвращённый `seq` с локальной записью → статус `delivered`.
5. UI: ⏱ → ✓.

**Гарантия:** переход в `delivered` означает, что шифротекст физически находится на сервере и доступен для pull получателем. Факт загрузки получателем этот статус не отражает.

***

### 3.3. READ — прочитано

#### 3.3.1. Принцип хранения

Сервер хранит единственное число — `last_seq` на пару `(username, dialogue_id)`. Это значение обновляется как побочный эффект каждого `pull`-запроса получателя по данному диалогу. Никакой таблицы соответствия `message → status` не существует ни на сервере, ни в базе.

Клиент-отправитель получает `partner_last_seq` и локально помечает все сообщения с `msg.seq <= partner_last_seq` как прочитанные.

#### 3.3.2. Эндпоинт `/arrived`

Единственный эндпоинт для чтения и управления статусом прочтения.

**GET — запрос текущего `last_seq` партнёра:**

```
GET /arrived?dialogue_id=<id>&partner=<username>
Authorization: serverid-signed header

Response 200:
{
  "partner_last_seq": <uint64 | null>,
  "ts": <unix_timestamp>
}
```

- `partner_last_seq: null` — получатель отключил уведомления о прочтении для этого диалога.
- Клиент вызывает этот эндпоинт только когда диалог открыт в UI. В фоне запросы не выполняются.

**PUT — управление настройкой получателем:**

```
PUT /arrived
Authorization: serverid-signed header

Body:
{
  "dialogue_id": "<id>",
  "receipts_enabled": true | false,
  "sig": "<Ed25519-подпись>"
}

Response 200: {}
```

#### 3.3.3. Когда клиент вызывает GET /arrived

- **Только при открытом диалоге.** Как только пользователь открывает диалог, клиент выполняет первый запрос.
- Повторные запросы — с интервалом, аналогичным notify polling (с jitter), но **только пока диалог остаётся открытым**.
- При закрытии диалога (сворачивание, переход в другой раздел) — опрос прекращается немедленно.
- В фоновом режиме (приложение свёрнуто, диалог не активен) — запросы не выполняются.

***

## 4. Управление функцией прочтения (per-dialogue opt-out)

### 4.1. Семантика opt-out

Отключение уведомлений о прочтении означает только одно: **сервер перестаёт обновлять `last_seq`** для данного диалога при pull-запросах получателя.

- Накопленное значение `last_seq` остаётся неизменным.
- Для отправителя это выглядит так, будто получатель перестал читать сообщения: ✓✓ больше не появляется на новых сообщениях, старые остаются с прежними статусами.
- Сервер не сообщает отправителю о факте отключения. Ответ `null` неотличим от «ещё не прочитано» для тех, кто никогда не запрашивал.

### 4.2. Поведение при изменении настройки

| Действие | Поведение |
|----------|-----------|
| Отключение | Сервер останавливает обновление `last_seq`; текущее значение сохраняется |
| Включение | Сервер возобновляет обновление `last_seq` при следующем pull получателя |
| Смена настройки | Не анонсируется отправителю; ответ `/arrived` без изменений внешне |

### 4.3. Управление настройкой в UI

- Настройка доступна per-dialogue: настройки конкретного диалога → «Уведомления о прочтении».
- Применяется немедленно после PUT /arrived.
- По умолчанию: `receipts_enabled = true`.

***

## 5. Хранение на сервере

### 5.1. Схема

```sql
CREATE TABLE dialogue_last_seq (
    username          TEXT    NOT NULL,
    dialogue_id       TEXT    NOT NULL,
    last_seq          INTEGER NOT NULL DEFAULT 0,
    receipts_enabled  BOOLEAN NOT NULL DEFAULT TRUE,
    updated_at        INTEGER NOT NULL,
    PRIMARY KEY (username, dialogue_id)
);
```

**Важно:** таблица не содержит никакой информации о конкретных сообщениях. `last_seq` — просто число. Вывод "какое сообщение прочитано" существует только в клиенте.

### 5.2. Серверная логика обновления `last_seq`

При каждом успешном `pull`-запросе от `username` по `dialogue_id`:

```
IF receipts_enabled[username][dialogue_id]:
    last_seq[username][dialogue_id] = MAX(current, pulled_seq)
```

`last_seq` монотонно возрастает и никогда не уменьшается.

***

## 6. Multi-device

- `last_seq` хранится на уровне аккаунта (`username`), не устройства.
- Если получатель прочёл диалог на устройстве B — `last_seq` обновляется, и отправитель увидит ✓✓ независимо от того, какое устройство получателя выполнило pull.
- Настройка `receipts_enabled` также единая для аккаунта per-dialogue. Расширение до per-device — возможная будущая версия.

***

## 7. Приватность и угрозы

### 7.1. Что сервер узнаёт

| Данные | Видимость серверу |
|--------|-------------------|
| Содержимое сообщения | ❌ E2E-шифрование |
| Факт отправки (sender, dialogue_id, seq) | ✅ metadata |
| Факт pull получателем + время | ✅ обновляет last_seq |
| Какое конкретно сообщение прочитано | ❌ только число last_seq |
| Вывод "это сообщение прочитано" | ❌ делает клиент |

### 7.2. Угроза: timing correlation через /arrived

Если клиент опрашивает `/arrived` только при открытом диалоге и с jitter — паттерн трафика идентичен обычной активности пользователя в диалоге. Фоновый опрос отсутствует — нет постоянного сигнала в трафике.

### 7.3. Угроза: inference через opt-out

Внезапная остановка обновления `last_seq` может теоретически сигнализировать об отключении. Контрмера — `null` в ответе неотличим от «не читал» при первом знакомстве, а сервер не сообщает отправителю об изменении настройки.

***

## 8. Edge-cases

| Случай | Поведение |
|--------|-----------|
| Получатель отключил receipts до получения сообщения | ✓✓ не появится; статус навсегда `delivered` |
| Сеть пропала после push, до pull-ack | `pending` → retry backoff → `delivered` после восстановления |
| Сообщение удалено через `cut_seq`/`determinate` | Статус у отправителя не меняется (локальное значение) |
| Получатель читает на устройстве B, не открывал на A | `last_seq` обновлён по аккаунту → ✓✓ у отправителя |
| Диалог закрыт до завершения опроса | Запрос /arrived отменяется; следующий — при открытии |
| Собеседник включил receipts после долгого отключения | Первый же pull обновит `last_seq`; отправитель увидит ✓✓ |

***

## 9. Сводка изменений в API

|----------|-------|-----------|
| `/arrived` | GET | Получить `partner_last_seq` для диалога |
