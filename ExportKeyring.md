# Опросник по шифрованному экспорту keyring

## Как я понял задачу

Экспорт нужен для переноса на новое устройство по A3b/F2b. Файл должен содержать не один активный ключ, а связку данных клиента: серверы, роли пользователя на этих серверах, диалоги и полный keyring каждого диалога. Данные критичны: утечка файла без защиты раскрывает приватные ключи пользователя/admin и все ключи диалогов, поэтому экспорт должен быть зашифрован и аутентифицирован.

Текущая реализация уже хранит локальный keyring диалога как список `{start_seq, key}` и использует его без изменения wire-format. Экспорт/импорт этого keyring в файл пока не реализован, потому что не выбраны параметры защиты экспорт-файла.

## Блок X — Состав экспорт-файла

### X1. Какие роли экспортировать?

- [ ] **X1a. Только клиентскую роль** — сервер URL, username, user private key, dialogs, keyring.
- [ ] **X1b. Клиентскую и admin-роли** — дополнительно admin private keys и список серверов, где есть admin-доступ.
- [ ] **X1c. Раздельные профили экспорта** — пользователь выбирает: client-only, admin-only или full.

### X2. Какие диалоги экспортировать?

- [ ] **X2a. Все диалоги всех серверов** — полный перенос устройства.
- [ ] **X2b. Только выбранный сервер** — перенос в рамках одного self-hosted инстанса.
- [ ] **X2c. Выбранные диалоги** — минимизация объёма и утечек при частичном переносе.

## Блок Y — Защита экспорт-файла

### Y1. Чем шифровать экспорт?

- [ ] **Y1a. Passphrase пользователя + Argon2id + XChaCha20-Poly1305** — удобный перенос файлом; нужна зависимость `argon2` и `chacha20poly1305` уже есть, но XChaCha API нужно проверить.
- [ ] **Y1b. Passphrase пользователя + Argon2id + AES-256-GCM** — стандартный AEAD, но потребуется новая зависимость AES-GCM.
- [ ] **Y1c. Экспорт шифруется публичным ключом принимающего устройства** — лучше против слабого пароля, но требует отдельного device-key протокола.

### Y2. Минимальные параметры passphrase/KDF?

- [ ] **Y2a. Только предупреждение UX** — пользователь сам отвечает за стойкую passphrase.
- [ ] **Y2b. Минимальная длина 12 символов + Argon2id interactive-параметры** — баланс UX и защиты.
- [ ] **Y2c. Минимальная длина 16 символов + Argon2id moderate-параметры** — медленнее, но лучше для чувствительного архива.

### Y3. Нужен ли recovery без passphrase?

- [ ] **Y3a. Нет** — потеря passphrase означает невозможность импортировать файл.
- [ ] **Y3b. Да, через локальное OS keychain/keystore** — удобнее, но хуже переносимость и требует платформенной реализации.

## Блок Z — Формат и UX

### Z1. Формат контейнера

- [ ] **Z1a. JSON envelope + base64 ciphertext** — проще отлаживать, больше размер.
- [ ] **Z1b. Бинарный envelope** — компактнее, сложнее UI/debug.

### Z2. Что делать при импорте конфликтующих данных?

- [ ] **Z2a. Merge по server+username+peer+start_seq** — добавлять отсутствующие ключи, совпадающие перезаписывать только если key равен.
- [ ] **Z2b. Replace profile** — импорт полностью заменяет локальные серверы/диалоги этого профиля.
- [ ] **Z2c. Preview перед импортом** — UI показывает, что будет добавлено/заменено.

### Z3. Удалять экспорт-файл после импорта?

- [ ] **Z3a. Только предупреждать пользователя** — приложение не управляет внешним файлом.
- [ ] **Z3b. Предлагать удалить файл после успешного импорта** — удобнее, но не гарантирует secure delete.

## Безопасное ограничение до ответов

До выбора X/Y/Z не нужно реализовывать экспорт приватных ключей, admin credentials или keyring в файл. Разрешён только локальный keyring внутри текущего устройства и ручной QR/JSON exchange, уже защищённый SAS-подтверждением.
