# Политика шифрования локального хранилища

---

## 1. Область применения

Настоящая политика определяет требования к шифрованию данных, хранящихся локально на устройстве пользователя в приложении Paranoia Messenger. Политика распространяется на все компоненты локального хранилища:

- конфигурационные файлы (`device_key.json`, `client.json`, `profiles.json`);
- файлы диалогов (`dialogs.json`);
- базу данных SQLite (`paranoia.db` и сопутствующие файлы WAL/SHM);
- кэш вложений (`attachment-cache/*`).

---

## 2. Принципы безопасности

Архитектура хранилища строится на следующих жёстких принципах:

1. **Нет доверия ОС.** Приложение не использует Android Keystore, iOS Secure Enclave или любые другие системные хранилища ключей. Безопасность не зависит от производителя устройства, ОС или сторонних прошивок.

2. **Нет биометрии.** Биометрическая аутентификация создаёт вектор принуждения («rubber hose attack») и требует доверия к TEE. Единственный фактор входа — знание PIN-кода.

3. **Master-ключ только в RAM.** `master_key` существует исключительно в оперативной памяти на время активной сессии. При закрытии приложения, переводе в фон (по истечении таймаута) или перезагрузке устройства ключ уничтожается. Повторный вход требует повторного ввода PIN.

4. **Нет резервного доступа.** Backup-коды, recovery-ключи и «секретные вопросы» не предусмотрены. Утеря PIN означает утрату доступа к данным. Это осознанный компромисс в пользу безопасности.

5. **Безопасность через математику.** Единственная защита от офлайн-атаки — энтропия PIN-кода и стоимость одной попытки перебора через Argon2id. Пользователь информируется об уровне защиты при выборе PIN.

---

## 3. Модель угроз

| Противник | Вектор атаки | Покрытие |
|---|---|---|
| Вор устройства | Физический доступ без PIN | ✅ Полное |
| Корпоративный шпионаж | Офлайн-атака на файлы хранилища | ✅ При PIN ≥ 10 цифр |
| Принуждение | Требование разблокировки под давлением | ✅ Нет биометрии → только PIN |
| Государственный forensic | Cellebrite, GrayKey и аналоги | ⚠️ Зависит от длины PIN |
| Компрометация ОС / TEE | Уязвимости системного хранилища | ✅ Не используется |
| Утеря PIN | — | ❌ Данные недоступны безвозвратно |

## 4. Пользовательский секрет и деривация мастер-ключа

### 4.1 PIN-код

Пользователь выбирает числовой PIN-код при первом запуске. Минимальная длина — **4 цифры**, рекомендуемая — **10 и более**.

При вводе интерфейс отображает **расчётное время взлома** при атаке GPU, чтобы пользователь осознанно выбрал баланс между удобством и уровнем защиты.

PIN **нигде не хранится** — ни в открытом виде, ни в хешированном.

**Надёжность по длине** (Argon2id m=64 МБ, t=3, ~100 попыток/сек на одном GPU):

| Длина | Комбинаций | Среднее время взлома | Уровень |
|---|---|---|---|
| 4 цифры | 10 000 | < 1 мин | ❌ Неприемлемо |
| 6 цифр | 1 000 000 | ~4 часа | ⚠️ Слабый |
| 8 цифр | 100 000 000 | ~16 суток | ⚠️ Средний |
| 10 цифр | 10 000 000 000 | ~4.5 года | ✅ Надёжный |
| 12 цифр | 1 000 000 000 000 | ~475 лет | ✅ Параноидальный |

> **Предупреждение:** пользователи склонны выбирать предсказуемые PIN-коды (даты рождения, повторения). Это многократно снижает реальную стойкость. Рекомендуется случайная генерация.

### 4.2 Функция деривации ключа — Argon2id

**Параметры:**

| Параметр | Значение | Обоснование |
|---|---|---|
| Алгоритм | Argon2id (RFC 9106) | Memory-hard, устойчив к GPU/ASIC |
| `m_cost` | 65536 KiB (64 МБ) | ~30 параллельных потоков на 16 ГБ VRAM |
| `t_cost` | 3 | OWASP 2025, баланс скорость/безопасность |
| `p_cost` | 1 | Стандарт для интерактивного входа |
| Длина ключа | 32 байта | AES-256 / ChaCha20-Poly1305 |
| Длина соли | 16 байт | CSPRNG, генерируется при создании профиля |

Соль хранится в открытом виде рядом с зашифрованными данными — секретом не является.

### 4.3 Деривация подключей — HKDF-SHA256

```
PIN
 └─► Argon2id(PIN, salt) ──► master_key [32B]  (только в RAM)
                                   │
              ┌────────────────────┼─────────────────────┐
              ▼                    ▼                      ▼
  HKDF("paranoia-json-v1")  HKDF("paranoia-db-v1")  HKDF("paranoia-files-v1")
         │                          │                       │
     json_key [32B]             db_key [32B]           files_key [32B]
```

Компрометация одного подключа не раскрывает остальные. При закрытии сессии все ключи обнуляются в памяти (`zeroize`).

## 5. Шифрование по типам данных

### 5.1 JSON-файлы

**Применяется:** `device_key.json`, `client.json`, `profiles.json`, `dialogs.json`

**Алгоритм:** ChaCha20-Poly1305 (RFC 8439)

**Ключ:** `json_key`

**Структура файла:**
```
[nonce: 12 байт] || [ciphertext + Poly1305 tag: N+16 байт]
```

**Верификация входа:** после расшифровки выполняется JSON-парсинг и проверка поля `"v"` (версия схемы). Ошибка парсинга или отсутствие поля `"v"` = неверный PIN. Отдельный хеш пароля не хранится.

**Почему ChaCha20-Poly1305:**
- Stream cipher — нет padding oracle уязвимостей
- Poly1305 защищает целостность данных (предотвращает bit-flip атаки на зашифрованный поток)
- Высокая производительность на мобильных CPU без аппаратного AES

### 5.2 База данных SQLite

**Применяется:** `paranoia.db`, `paranoia.db-shm`, `paranoia.db-wal`

**Алгоритм:** SQLCipher — AES-256-CBC + HMAC-SHA512

**Ключ:** `db_key`, передаётся как raw-ключ: `PRAGMA key = "x'...'"`

**Параметры SQLCipher:**

| PRAGMA | Значение | Примечание |
|---|---|---|
| `cipher_page_size` | 4096 | Стандартный размер страницы |
| `kdf_iter` | 1 | Внешний KDF (Argon2id), внутренний отключён |
| `cipher_hmac_algorithm` | HMAC_SHA512 | SQLCipher 4 по умолчанию |

`kdf_iter = 1` обязателен: без него SQLCipher запускает собственный PBKDF2 поверх готового ключа — избыточно и некорректно.

### 5.3 Кэш вложений

**Применяется:** `attachment-cache/*`

**Алгоритм:** ChaCha20-Poly1305

**Ключ:** уникальный per-file дериват через HKDF:
```
file_key = HKDF(files_key, salt=attachment_uuid_bytes, info="attachment-v1")
```

Структура файла: `[nonce: 12B] || [ciphertext + tag]`

Оригинальное расширение хранится в метаданных БД, а не в имени файла.

## 6. Жизненный цикл master_key в памяти

```
Запуск приложения
      │
      ▼
  Ввод PIN
      │
      ▼
  Argon2id ──► master_key (heap, зафиксирован в RAM, mlock)
      │
      ▼
  Открыта сессия
      │
      ├── Приложение в фоне > N минут ──► zeroize(master_key) ──► экран блокировки
      ├── Явный выход пользователя    ──► zeroize(master_key)
      └── Перезапуск / перезагрузка   ──► master_key не существует
```

**Меры защиты ключа в RAM:**
- `mlock()` — запрет выгрузки страниц с ключом в swap
- `zeroize` (крейт) — гарантированное обнуление при drop (защита от оптимизатора компилятора)
- Ключ не копируется без необходимости, не логируется, не сериализуется

## 7. Аутентификация и блокировка

### 7.1 Вход

1. Пользователь вводит PIN
2. `Argon2id(PIN, salt)` → `master_key` (в RAM)
3. HKDF → `json_key`, `db_key`, `files_key`
4. Расшифровка `client.json` → JSON-парсинг → проверка поля `"v"`
5. Успех: открытие SQLite с `db_key`, сессия активна
6. Неудача: инкремент счётчика, задержка по политике блокировки

### 7.2 Политика блокировки

| Попытки | Действие |
|---|---|
| 1–5 | Немедленный отказ |
| 6–10 | Задержка 30 секунд |
| 11–15 | Задержка 5 минут |
| 16–20 | Задержка 30 минут |
| > 20 | Полная блокировка до перезапуска приложения |

### 7.3 Смена PIN

1. Успешный вход со старым PIN → `master_key` в RAM
2. Генерация новой соли (CSPRNG)
3. `Argon2id(new_PIN, new_salt)` → новый `master_key`
4. Перешифровка всех файлов с новыми подключами
5. Уничтожение старых данных, `zeroize` старого `master_key`

---

## 8. Удаление профиля

При удалении профиля:
1. `zeroize` всех ключей в RAM
2. Однопроходная перезапись файлов случайными байтами
3. Удаление файлов и директории профиля

На устройствах с NAND (SSD/eMMC) гарантированное физическое уничтожение невозможно из-за wear-leveling. Это приемлемо: без `master_key` зашифрованные данные криптографически недоступны.

## 9. Сводная таблица алгоритмов

| Компонент | Алгоритм | Ключ | Хранение ключа |
|---|---|---|---|
| Деривация мастер-ключа | Argon2id (m=64MB, t=3) | PIN + соль | **Только RAM** |
| Деривация подключей | HKDF-SHA256 | master_key | **Только RAM** |
| JSON-файлы | ChaCha20-Poly1305 | json_key | **Только RAM** |
| SQLite БД | AES-256-CBC (SQLCipher) | db_key | **Только RAM** |
| Файлы вложений | ChaCha20-Poly1305 | per-file HKDF | **Только RAM** |

## 10. Реализация в Rust

**Крейты:**

| Функция | Крейт |
|---|---|
| Argon2id | `argon2` 0.5+ |
| ChaCha20-Poly1305 | `chacha20poly1305` 0.10+ |
| HKDF | `hkdf` + `sha2` 0.12+ |
| SQLCipher | `rusqlite` (feature `bundled-sqlcipher`) 0.31+ |
| Обнуление памяти | `zeroize` 1.7+ |
| CSPRNG | `rand` (feature `getrandom`) 0.8+ |

**Пример:**

```rust
use argon2::{Argon2, Algorithm, Version, Params};
use hkdf::Hkdf;
use sha2::Sha256;
use zeroize::Zeroizing;

fn derive_master_key(pin: &str, salt: &[u8; 16]) -> Zeroizing<[u8; 32]> {
    let params = Params::new(65536, 3, 1, Some(32)).unwrap();
    let argon2 = Argon2::new(Algorithm::Argon2id, Version::V0x13, params);
    let mut key = Zeroizing::new([0u8; 32]);
    argon2.hash_password_into(pin.as_bytes(), salt, key.as_mut()).unwrap();
    key // zeroize автоматически при drop
}

fn derive_subkey(master: &[u8; 32], context: &[u8]) -> Zeroizing<[u8; 32]> {
    let hk = Hkdf::<Sha256>::new(None, master);
    let mut out = Zeroizing::new([0u8; 32]);
    hk.expand(context, out.as_mut()).unwrap();
    out
}
```
