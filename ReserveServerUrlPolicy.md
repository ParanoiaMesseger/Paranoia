# Политика резервных URL-адресов сервера

Система поддерживает настройку резервных адресов доступа к серверу для обеспечения отказоустойчивости и возможности маршрутизации трафика через CDN или промежуточные прокси без изменений серверного кода.

***

## Принципы транспортного fallback

Клиент хранит упорядоченный список эндпоинтов: основной URL и один или несколько резервных (`reserve_server_urls`). При каждом запросе клиент перебирает список по порядку и использует первый успешно ответивший эндпоинт. Все эндпоинты обращаются к одному и тому же origin-серверу — семантика запросов не меняется. Резервные URL хранятся раздельно для admin-профиля и client-профиля.

***

## Алгоритм выбора эндпоинта

Список `ENDPOINTS` формируется динамически из сохранённых профильных данных. Клиентская FFI-инициализация и admin-регистрация получают `reserve_server_urls`, и fallback-логика применяется единообразно ко всем сетевым операциям.

***

## Эталонная CDN-архитектура

Резервный путь реализуется через DNS + CDN-провайдер без изменений на origin-сервере:

```
Прямой путь:
  Клиент → origin.example.com (A → IP) → Nginx → Axum

CDN-путь (резервный):
  Клиент → cdn.example.com (CNAME → CDN) → CDN Edge
         → origin.example.com → Nginx → Axum
```

Параметры CDN-ресурса: origin — `origin.example.com`, протокол — HTTPS, кэширование — отключено, разрешённые методы — `GET, PUT, HEAD, OPTIONS`.

***

## Требования к TLS

Origin-сервер должен принимать TLS-соединения по обоим именам. Сертификат охватывает оба домена — основной и CDN-алиас. Nginx указывает оба домена в `server_name`. Альтернатива — wildcard-сертификат. TLS-терминация на CDN-крае выполняется отдельным сертификатом, выпущенным через Certificate Manager CDN-провайдера.

***

## Безопасность и модель доверия

- Криптографические гарантии не зависят от маршрута: подписи Ed25519 и структура пакетов остаются неизменными вне зависимости от того, через какой эндпоинт прошёл запрос.
- CDN-нода видит зашифрованный TLS-трафик; при сквозном TLS (origin — HTTPS) содержимое запросов CDN не раскрывается.
- Компрометация резервного домена позволяет злоумышленнику наблюдать метаданные соединений, но не содержимое пакетов — они защищены end-to-end шифрованием прикладного уровня.
- Список резервных URL не верифицируется криптографически: ответственность за корректность конфигурации лежит на администраторе.

***

## Ограничения

- Система не выполняет автоматического health-check эндпоинтов в фоне — переключение происходит только в момент отказа конкретного запроса.
- При одновременном отказе всех эндпоинтов операция возвращает ошибку без повторных попыток на транспортном уровне.
- Конфигурация DNS, CDN, TLS и Nginx находится полностью вне области ответственности приложения.
