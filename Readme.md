# Paranoia — Требования по безопасности и обоснование решений

## 1. Модель угроз и цели

### 1.1. Нарушить

Paranoia проектируется против сильного нарушителя, который:

- Контролирует каналы связи (провайдер, государственный цензор) и может:
  - выполнять пассивное наблюдение за трафиком (полная запись);
  - осуществлять активные атаки (MITM, модификация/повтор, блокировка);
  - выпонять несанкционированный выпуск tls - сертификатов сервера;
  - применять DPI/ML-анализ трафика.
- Может компрометировать сервер:
  - получить доступ к хранилищу (диску, базе данных);
  - получить доступ к логам и метрикам;
  - модифицировать серверный код.
- Может скомпрометировать отдельные клиентские устройства:
  - получить доступ к локальному хранилищу (БД с историей);
  - подменить доверенные ssl сертификаты на устройстве;
  - запустить вредоносное ПО на устройстве.

### 1.2. Цели безопасности

Paranoia стремится обеспечить:

- Конфиденциальность:
  - содержимое сообщений недоступно ни серверу, ни MITM на уровне транспорта;
  - для компрометированного сервера данные остаются криптографически защищёнными.
- Целостность:
  - противник не может незаметно модифицировать сообщения или состояние диалога;
  - любые подделки или попытки переиграть историю либо очевидны, либо невозможны.
- Аутентичность:
  - получатель может убедиться, что сообщение создано конкретным отправителем.
- Управляемость истории:
  - пользователь может инициировать удаление своих данных с сервера (в рамках модели диалога);
  - сервер не может «подсовывать» несуществующую историю от имени пользователя.
- Повышенную скрытность:
  - протокол не очевиден при простом анализе трафика;
  - при раскрытии текущей маски сервер/клиент могут перейти на другую маску без смены криптографического ядра.

## 2. Защита на уровне хранилища и сервера

### 2.1. Zero-trust сервер

**Цель:** даже полностью скомпрометированный сервер не должен раскрывать содержимое сообщений и ключи.

Решения:

- Сервер хранит только зашифрованные payload’ы:
  - все сообщения шифруются на клиенте симметричным ключом диалога;
  - сервер не обладает ключами шифрования/дешифрования;
  - процесс обмена ключами не проходит через сервер.
- Сервер не хранит приватные ключи подписей:
  - хранится только публичный ключ администратора и публичные ключи пользователей.

**Защита:**

- От утечки базы данных сервера:
  - атакующий видит только шифротексты и минимальные метаданные (seq, dialogue_id).
- От злонамеренного/компрометированного оператора:
  - оператор не может читать сообщения или подменять их содержимое незаметно (из-за подписей и end-to-end шифрования).

### 2.2. Dialogue-oriented storage

**Цель:** обеспечить целостную модель диалогов и восстановление истории, не раскрывая лишнюю информацию.

Решения:

- Хранение по диалогам, а не по inbox-пользователя:
  - диалог — минимальная единица истории;
  - идентификатор диалога вычисляется как хеш от пары участников (симметрично).
- Для записи используется ключ вида:
  - `dialogue_id:seq`, где `seq` — монотонный номер.

**Защита:**

- От нарушения целостности истории:
  - история диалога A <=> B хранится в одном непрерывном пространстве seq;
  - невозможно «разнести» часть диалога в другое логическое место.
- От раскрытия отправителя в хранилище:
  - серверу достаточно знать пару участников диалога (через хеш), но не нужно хранить отправителя отдельно для каждого сообщения;
  - отправитель определяется на клиенте по внутреннему заголовку.

Стоимость:

- Сервер «узнаёт», что пользователи A и B общаются (по факту существования диалога), но не знает, кто из них инициировал конкретное сообщение.

### 2.3. Защита от перезаписи сообщений

**Цель:** избежать атак, при которых злоумышленник повторно отправляет пакет с тем же `seq`, чтобы повлиять на историю (replay/overwrite).

Решения:

- Запрет перезаписи ключа в хранилище:
  - перед записью `dialogue_id:seq` сервер проверяет, существует ли сообщение;
  - если сообщение есть - операция отклоняется (дубликат seq).
- уникальный `seq` в рамках диалога:
  - формирование со стороны клиента (локальный счётчик)
  - проверка корректности на сервере.

**Защита:**

- От атак типа replay внутри диалога:
  - нелзья переписать сообщение за счет повторной отправки пакета.
  - без приватного ключа отправителя нелзья отправить сообщение с другим `seq` т.к. подпись не совпадет.
- От устаревшего состояния клиента:
  - если клиент с «старым состоянием» пытается отправить `seq`, уже использованный другой копией клиента, сервер это обнаружит.

## 3. Защита целостности и аутентичности

### 3.1. Подписи и проверка отправителя

**Цель:** гарантировать авторство сообщения и предотвратить модификацию.

Решения:

- Использование схемы Ed25519:
  - каждый пользователь имеет свой ключ подписи;
  - подпись включает все тело запроса (Например запрос push Ed25519_SIGN({username}{partner}{seq}{payload_b64}))
  - сервер хранит username <-> публичный ключ. Проверка подписи идет по ключу для username.

**Защита:**

- От подделки содержания: изменение payload приводит к неверной подписи.
- От подделки участников диалога: нельзя изменитьимена участников без приватного ключа отправителя.
- От подмены управления историей (`pull`/`determinate`): операции на историю (загрузка, удаление) нельзя выполнить без подписи одного из участников диалога.

### 3.2. Право на забвение

**Цель:** дать пользователю контроль над своей историей, не ломая модель zero-trust.

Решения:

- Удаление по диалогу (determinate):
  - операция удаляет все сообщения диалога с `seq <= cut_seq`;
  - инициировать может любой участник диалога без согласия собеседника.
- Идемпотентность:
  - повторный вызов с тем же или меньшим `cut_seq` не приводит к некорректным состояниям.

**Защита:**

- От накопления компрометирующей истории: участник может удалить историю диалога с сервера (в полном объеме до cut_seq).
- От несанкционированного удаления третьими лицами: требуется валидная подпись участника диалога.

Стоимость:

- Один участник может удалить историю диалога. Это модель ставит в приоритет «право на забвение».

## 4. Защита на уровне транспорта и трафика (скрытность)

### 4.1. Независимость от TLS-секретности

**Цель:** Zero Trust к TLS как единственному уровню защиты, так как MITM на TLS возможен (например, цензурный прокси).

Решения:

- Симметричное END-TO-END шифрование:
  - сообщения клиентов симметрично зашифрованы (ChaCha20-Poly1305);
  - используется модель с предварительно распределёнными ключами;
  - ключи не формируются через это соединение;
  - раскрытие TLS-ключей или подмена сертификатов не раскрывает содержимое сообщений.
- Подписи поверх зашифрованного payload:
  - целостность и аутентичность сохраняются даже при активном MITM на TLS.
  - при компрометации подписи - END-TO-END шифрование все равно защитит от модификации сообщения.

### 4.2. Маскировка формата и размера сообщений

**Цель:** снизить вероятность того, что DPI/цензор однозначно идентифицирует протокол по размерам и структуре пакетов.

Решения:

- Рандомизация размеров чанков при отправке файлов:
  - файл делится на чанки случайного размера в заданном диапазоне;
  - каждое сообщение-чанк имеет непредсказуемый размер.
- Дополнительный паддинг внутри зашифрованного payload’а:
  - произвольное количество «мусорных» байтов, удаляемых после расшифровки;
  - уменьшает корреляцию между размером полезной нагрузки и размером пакета.

**Защита:**

- От size-based анализа:
  - сложнее определить тип сообщений (текст/файл/голос) по размеру пакетов;
  - сложнее оценить размер передаваемых файлов.

Стоимость:

- Увеличение объёма передаваемых данных.
- Снижение скорости обена сообщениями.

### 4.3. Маскарадный JSON-слой (cover layer)

**Цель:** скрыть сам протокол Paranoia от простого анализа HTTP/JSON-трафика, дать возможность быстро менять «облик» API.

Решения:

- Введение модульного маскарадного слоя:
  - ядро Paranoia работает с внутренними структурами (CorePush/CorePull/CoreDeterminate);
  - всего одна прослойка (cover module) преобразует их в «правдоподобный» JSON и обратно.
- Базовый модуль маскарада:
  - JSON выглядит как API бизнес-сервиса (например, доставка еды, CRM и т.п.);
  - внутренние данные (sender, recver, seq, payload, sig) сериализуются в бинарный формат и распыляются по нескольким полям JSON (например, `meta.p1`, `meta.p2`, `meta.p3`, `meta.p4`), выглядящим как обычные строки/идентификаторы.
- Модульность:
  - cover реализован как сменный модуль на клиенте и сервере;
  - смена маски = смена реализации cover без изменения криптоядра и хранилища.

**Защита:**

- От сигнатурного детекта на уровне HTTP:
  - внешний вид JSON не содержит очевидных полей протокола (sender/payload/sig);
  - разные инстансы могут использовать разные маски (разные схемы JSON).
- От целенаправленного расследования:
  - при компрометации одной маски её можно заменить другой, не трогая низкий уровень.

Стоимость:

- Усложнение кода (двойная сериализация, модульность);
- Риск ошибок в cover-реализации, если она написана неаккуратно.
- Увеличение оверхеда, стоймости отправки сообщения.

## 5. Возможные угрозы

Несмотря на вышеописанные меры, Paranoia имеет ряд компромиссов, на которые пришлось пойти ради удобства или скорости разработки.

1. **Хранение истории:**
   - Сервер хранит историю, пусть и в зашифрованном виде.
   - История - сознательный компромисс в пользу доступности.
   - При использовании надежных каналов для обмена ключами, дешифровка истории невозможна.

2. **Компрометация клиентского устройства:**
   - если злоумышленник получает полный доступ к устройству (malware, физический доступ):
     - он может прочитать локальную историю, ключи, сообщения в открытом виде;
     - протокол Paranoia здесь не поможет — необходимы отдельные OS-level меры (шифрование диска, sandboxing и т.п.).

3. **Глобальный трафик-анализ на уровне метаданных:**
   - скрытность снижает вероятность точного обнаружения, но не исключает:
     - анализ корреляций по IP/домену/времени;
     - анализ суммарного объёма трафика;
     - сопоставление активности с другими событиями.
   - полностью скрыться как «обычный веб-сёрфинг» невозможно без более тяжёлых механизмов (mix-сети, Project-X (X-RAY)).

4. **Комплексный DPI с ML и длительным обучением:**
   - даже с маскарадом и паддингом, специализированные системы могут со временем выучить паттерны поведения приложений и выделить Paranoia;
   - проект ориентирован на повышение стоимости детекции, а не на абсолютную неотличимость.
   - cover можно усложнить за счет реализации через Json-SHEMA, что снизит стоймость разработки нового маскарада, но не снизит стоймость разработки детектора.

5. **Сильный активный цензор, блокирующий домены/IP:**
   - если цензор решает блокировать домен/адрес целиком, Paranoia не может помешать этому, но SelfHosted-подход позволит быстро мигрировать на другой хост с новым маскарадом.
   - можно использовать внешний слой обхода (ротация доменов, прокси, DNS-трюки, распределённая инфраструктура).

6. **Ответственность за приватность ключей:**
   - утечка приватных ключей пользователей возможна только на стороне клиента (через фишинг, слабые пароли, бэкапы в облаке). Такая утечка обнуляет гарантии для этих пользователей.
   - протокол не защищает от ошибок пользователя при хранении ключей.

7. **Неанонимность при компрометации сервера:**
   - Paranoia не может защитить от анализа IP клиентоа:
   - Злонамеренно модифицированный сервер может накопить информацию об ip участников диалога.
   - Для защиты ip нужно применять другие технологии, VPN, proxy.

8. **Атаки на реализацию:**
   - требования описывают логическую модель безопасности, но возможны ошибки коде (в криптопримитивах, побочные каналы, утечки памяти);
   - необходимы аудит кода, тестирование и, по возможности, формальные проверки критичных частей.

---

Эти требования задают рамки того, **от чего** Paranoia должна защищать, **какими механизмами** достигаются эти цели и **какие риски остаются в зоне ответственности инфраструктуры, оператора и конечного пользователя.**
