# Paranoia — Политика реализации системы уведомлений

## Обзор

Данный документ описывает архитектурные решения, протокол и платформенную реализацию системы уведомлений мессенджера Paranoia. Все решения приняты в рамках действующей модели угроз: сервер считается потенциально скомпрометированным, Google/Apple не являются доверенными сторонами, self-hosted развёртывание является основным режимом работы.

---

## 1. Принципы системы уведомлений

### 1.1. Что сервер не должен знать

- Количество устройств пользователя
- Платформу и тип устройства (iOS, Android, Desktop)
- Push-токены (FCM, APNs) — они не используются
- От кого пришло сообщение и его содержимое
- Факт прочтения сообщения

### 1.2. Что сервер неизбежно знает

- IP-адрес клиента в момент polling-запроса
- Время запроса
- Идентификатор диалога и последний известный клиенту `seq` (передаётся в `/notify`)

Эти метаданные признаются допустимым компромиссом, задокументированным в разделе 5.3 модели угроз (корреляция по IP/времени).

### 1.3. Отказ от FCM и APNs

FCM и APNs не используются по следующим причинам:
- Self-hosted развёртывание не имеет Developer-сертификата Apple
- Google и Apple становятся наблюдателями в цепочке доставки
- FCM/APNs требуют хранения device-токенов, связанных с аккаунтом

## 2. Серверный протокол — эндпоинт `/notify`

### 2.1. Место в API

К существующим эндпоинтам `/reg`, `/push`, `/pull`, `/determinate` добавляется новый:

```
POST /notify
```

### 2.2. Формат запроса

Запрос аналогичен `/pull` по структуре подписи. Клиент передаёт идентификатор диалога и последний известный `seq`:

```json
{
  "sender":   "alice",
  "partner":  "bob",
  "seq":      42,
  "sig":      "<Ed25519_SIGN(sender||partner||seq)>"
}
```

Подпись покрывает `sender`, `partner`, `seq` — аналогично существующим операциям. Сервер проверяет подпись по публичному ключу `sender`.

### 2.3. Формат ответа

```json
{ "n": 3 }
```

Где `n` — количество сообщений с `seq` больше переданного до конца текущей очереди диалога. Если `n = 0` — новых сообщений нет. Сервер не передаёт содержимое, идентификаторы отправителей или любые другие метаданные.

### 2.4. Cover-слой

Запрос и ответ проходят через маскарадный cover-слой (раздел 4.3 модели угроз) наравне с остальными эндпоинтами. Внешний вид JSON-полей определяется активным cover-модулем.

## 3. Клиентский механизм — randomized polling

### 3.1. Алгоритм

Клиент запускает цикл polling с псевдослучайным интервалом:

```
interval = random(10, 240) секунд
```

Случайный разброс от 10 до 240 секунд преследует две цели:
- Снижает корреляцию между отправкой сообщения и моментом запроса `/notify` — затрудняет traffic correlation attack
- Снижает предсказуемость паттерна запросов при DPI-анализе

### 3.2. Поведение при получении `n > 0`

При `n > 0` клиент **не загружает сообщения немедленно**. Показывается только системное уведомление:

```
"Новых сообщений: N"
```

Без имени отправителя, без текста, без идентификатора диалога. Содержимое подгружается через `/pull` только когда пользователь открывает конкретный диалог.

### 3.3. Поведение при `n = 0`

Планируется следующий polling-запрос через новый случайный интервал. Никаких дополнительных действий.

### 3.4. Reconnect-стратегия

При потере соединения или ошибке сети — exponential backoff с jitter:

```
delay = min(1000 * 2^retry_count, 60000) мс
delay += random(0, delay * 0.2) мс  // jitter ±20%
```

При обнаружении смены сети через системное API (`QNetworkInformation`) — счётчик retry сбрасывается, переподключение немедленное.

## 4. Платформенная реализация

### 4.1. Linux и Windows — системный трей

Приложение работает в системном трее после закрытия основного окна. Polling выполняется в фоновом потоке штатными средствами Qt без ограничений ОС.

- **Уведомления:** `QSystemTrayIcon::showMessage()` — нативные уведомления (Action Center на Windows, libnotify/D-Bus на Linux)
- **Фоновый процесс:** обычный `QTimer` в отдельном потоке
- **Поведение при закрытии окна:** `closeEvent` переопределяется — приложение скрывается в трей, процесс не завершается

### 4.2. Android — Foreground Service

Polling выполняется в Android Foreground Service, который устойчив к battery optimization и Doze mode.

- **Индикатор:** системное уведомление в статус-баре (обязательное требование Android 8+)
- **Текст плашки:** "Paranoia: ожидание сообщений" — прозрачно для пользователя
- **Реализация в Qt:** `QAndroidService` + Java Foreground Service
- **Разрешения в манифесте:** `FOREGROUND_SERVICE`, `FOREGROUND_SERVICE_DATA_SYNC`
- **Смена сети:** отслеживается через `ConnectivityManager`, при смене — немедленный новый запрос

### 4.3. iOS 26+ — BGContinuedProcessingTask

На iOS 26 и выше используется новый API фоновых задач, представленный на WWDC 2025, который позволяет держать долгоживущий фоновый процесс без VPN-хака.

- **Идентификатор задачи:** `com.paranoia.polling`
- **Поведение:** задача регистрируется при старте, самостоятельно перепланируется при каждом запуске
- **Реализация:** Objective-C++ (`.mm`), вызывается из Qt через `extern` функцию
- **Смена сети:** `NWPathMonitor` → немедленный новый запрос

### 4.4. iOS < 26 — без фоновых уведомлений

На iOS версий ниже 26 фоновые уведомления **не реализуются**. Это явный, задокументированный компромисс в пользу модели угроз.

**Причины отказа от альтернатив:**
- `NEPacketTunnelProvider` (VPN-хак): конфликтует с реальными VPN-сервисами пользователя — неприемлемо для целевой аудитории
- `BGAppRefreshTask`: ненадёжен в реальных условиях, интервал не гарантирован, контролируется ОС
- APNs: требует серверного хранения device-токена — нарушает модель угроз

**Информирование пользователя:** при первом запуске на iOS < 26 отображается явное сообщение:

```
⚠️ Фоновые уведомления недоступны

На iOS 25 и ниже Paranoia не может проверять
новые сообщения в фоне — это цена полной
приватности без использования серверов Apple.

Уведомления работают только пока
приложение открыто.

Для фоновых уведомлений требуется iOS 26+.
```

## 6. Угрозы, которые система не устраняет

### 6.1. Traffic correlation attack

Наблюдатель, контролирующий трафик обоих участников, может сопоставить момент отправки сообщения одним с моментом получения `n > 0` другим. Случайный интервал polling (10–240 с) **снижает**, но не исключает эту корреляцию.

**Частичное решение на сервере:** случайная задержка ответа `/notify` на 0–N секунд (jitter на стороне сервера) дополнительно разрывает корреляцию.

### 6.2. IP-раскрытие

Polling-запросы к серверу раскрывают IP клиента и факт активности. Для защиты требуются внешние средства: VPN, Tor, Project-X (XRAY) — вне scope данного документа, задокументировано в разделе 5.7 модели угроз.

### 6.3. Анализ частоты запросов

Специализированные DPI-системы с ML могут выучить паттерн случайных polling-запросов. Cover-слой (раздел 4.3) снижает стоимость детекции, но не устраняет угрозу полностью.

---

## 7. Допущения и компромиссы

- **Задержка уведомления:** от 10 до 240 секунд — приемлемо для целевой аудитории с учётом российских реалий доступа к интернету
- **Дополнительный round-trip:** `/notify` добавляет нагрузку на сервер при большом числе клиентов; при необходимости — rate limiting на эндпоинт
- **iOS < 26:** отсутствие фоновых уведомлений — явный и задокументированный компромисс, сообщаемый пользователю при установке
